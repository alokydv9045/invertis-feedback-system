# 🚀 Render Deployment Guide: Invertis Feedback System

This guide outlines how to deploy both the frontend and backend of the **Invertis Feedback System** on **Render** using a single, unified blueprint configuration.

## 🛠️ Step 1: Database Setup Options

Since the project uses **PostgreSQL** with Prisma, you need a database connection string. You have two options:

### Option A: Use Render's Managed PostgreSQL (Default & Automated)
The included root `render.yaml` automatically provisions a free Render PostgreSQL database (`tlfq-db`) and binds its connection string to the backend service.
- **Pros**: Completely automated, zero configuration required.
- **Cons**: Render's free tier databases expire and are deleted after **90 days**.

### Option B: Use an External PostgreSQL Database (Recommended for Production)
You can create a free, permanent PostgreSQL database using a provider like [Supabase](https://supabase.com) or [Neon](https://neon.tech).
1. Create a database project.
2. Retrieve your connection string (e.g. `postgresql://...`).
3. You will override the `DATABASE_URL` environment variable in the Render Dashboard when deploying the backend.

---

## 🌐 Step 2: One-Click Deploy using Render Blueprints

Render Blueprints let you deploy your entire stack—frontend, backend, and database—with a single setup.

1. Commit and push all code changes to your GitHub repository:
   ```bash
   git add .
   git commit -m "Configure unified Render blueprint deployment"
   git push origin main
   ```
2. Go to your [Render Dashboard](https://dashboard.render.com).
3. Click the **New +** button in the top right and select **Blueprint**.
4. Connect your GitHub repository containing the project.
5. Render will automatically detect the root `render.yaml` file and show a list of services to be created:
   - **`tlfq-db`**: PostgreSQL Database (if using Option A).
   - **`tlfq-api`**: Node.js Backend Web Service.
   - **`tlfq-frontend`**: Static Site Frontend.
6. Provide a Blueprint Group Name (e.g. `invertis-feedback-system`).
7. Complete any required environment variables:
   - **`JWT_SECRET`**: Automatically generated by Render (or enter a custom secure string).
   - **`DATABASE_URL`**: Leave blank if using Option A (Render DB). If using Option B (Supabase/Neon), enter your external database connection string to override the default.
8. Click **Apply**.

## 🔐 Environment Variables Reference

### Backend (`tlfq-api`)
These variables are set in the `render.yaml` blueprint or can be adjusted in the Render dashboard under **Service Settings** -> **Environment**:
* `NODE_ENV`: `production`
* `DATABASE_URL`: Connection string to PostgreSQL.
* `JWT_SECRET`: A secure key to sign authentication tokens.
* `CORS_ORIGIN`: `https://invertis-feedback-system-2.onrender.com` (Allows the frontend to communicate with the API).
* `SEED_DATA`: `true` (Enables automatic creation of initial academic sessions, departments, and default admin accounts on startup).

### Frontend (`tlfq-frontend`)
* `VITE_API_URL`: `https://invertis-feedback-system-0chx.onrender.com` (Tells the frontend SPA where to send API requests).

## 🚨 Troubleshooting & Common Issues

### 1. Frontend Routing Returns 404 on Refresh
* **Cause**: Client-side router URLs (like `/dashboard`) do not exist as physical folders on a static host.
* **Solution**: The root `render.yaml` already has a rewrite rule configured (`source: /*` -> `destination: /index.html`). Make sure this rule is present in your service configuration under the **Redirects/Rewrites** section of the frontend Static Site in the Render Dashboard.

### 2. CORS Blocked Errors in Browser Console
* **Cause**: The backend is rejecting requests because the origin does not match.
* **Solution**: Verify that `CORS_ORIGIN` in the backend service environment variables is set to exactly `https://invertis-feedback-system-2.onrender.com` (no trailing slash).

### 3. Prisma Schema Out of Sync
* **Cause**: Database tables are missing or not updated.
* **Solution**: The blueprint build command is configured as `npm install && npx prisma generate && npx prisma db push`. The `db push` command ensures that the PostgreSQL database tables are automatically kept in sync with the Prisma schema code on every deployment.
